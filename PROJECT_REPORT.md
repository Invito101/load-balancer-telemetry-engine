## 1. Problem Statement

Build a simple load-balancing system to demonstrate core OS concepts: concurrency control with mutexes, synchronization/back-pressure using semaphores, file locking for durable writes, inter-process communication (TCP sockets and a named FIFO), and role-based access control. The system accepts job requests from clients and forwards them to backend worker processes.


## 2. High-level Design

- A load-balancer server listens on a TCP port and accepts client connections.
- Backend workers (worker1, worker2) listen on their own TCP ports and process `JOB` requests.
- An audit logger reads a named FIFO written by the server to record events.
- Data files in `data/` seed users and backends and store request logs.
- The client is interactive and prompts for username, password, and a command.

Components:
- `bin/lb_server` ‚Äî load balancer server
- `bin/backend_worker` (symlinked as `bin/worker1`, `bin/worker2`) ‚Äî worker process
- `bin/lb_client` ‚Äî interactive client
- `bin/audit_logger` ‚Äî reads FIFO and appends to audit log
- `scripts/` ‚Äî demo and test scripts


## 3. Required OS Concepts and Implementation

**Mutexes (pthread mutexes)**
- Where used: `src/backend_worker.c` (`active_requests`) and `src/lb_server_state.c` (protects backend array and per-backend counters).
- Purpose: Prevent race conditions when multiple threads update shared counters.

**Semaphores (POSIX semaphores)**
- Where used: `src/lb_server.c` via `job_slots`.
- Purpose: Limit in-flight forwarded jobs (back-pressure). Hardcoded limit: 2.

**File locking (`flock`)**
- Where used: `src/common.c::append_line_locked()` and `src/lb_server_state.c::load_backends()`.
- Purpose: Ensure `data/requests.db` and `data/backends.db` are updated or read atomically across processes.

**Sockets (TCP)**
- Server listens for clients; server forwards `JOB` requests to workers over TCP.
- Worker accepts connections and replies with `OK <worker> at <timestamp> completed <payload>`.

**IPC (named pipe / FIFO)**
- Server writes audit events to `run/lb_events.fifo`.
- `audit_logger` reads from the FIFO and appends time-stamped entries to `data/audit.log`.

**Role-based authorization**
- Two roles only: `admin` (can `LIST_BACKENDS`) and `user` (can `JOB`).
- Credentials in `data/users.db` (seed):
  - `admin:admin123:admin`
  - `alice:alice123:user`

## 8. Challenges Faced & Solutions

- Concurrency nondeterminism in tests: The mutex test depended on routing; fixed by forcing test to target a single backend so concurrency is observable.
- Simplifying configuration: Migrated from `config.txt` to hardcoded defaults (to match project constraints) and removed CLI flags; needed careful pruning of config readers.
- Keeping demos consistent: Updated scripts and README after refactoring to ensure reproducible demo/test runs.


## 9. Design Decisions & Trade-offs

- Hardcoded defaults (host/ports, FIFO path, default sleep) simplify demos and grading but reduce configurability.
- Removed guest role and admin/user split reduces complexity and focuses assessment on OS concepts.
- Backend persistence (write) was removed to make backends read-only seeds; this simplifies state management for the demo.
